Add a new repo to watch

To watch a new job-aggregator repo (e.g. when the Summer 2027 Simplify repo drops, or you want to add speedyapply/2026-SWE-College-Jobs):

  1. Open .github/workflows/watch-files.yml in your editor.
  2. Find the WATCHERS list near the top of the Python heredoc (around line 35).
  3. Add a new entry to the list. Example for a HTML-table repo: 
    {
    "label": "Speedy Apply",
    "owner": "speedyapply",
    "repo": "2026-SWE-College-Jobs",
    "branch": "main",
    "file": "README.md",
    "parser": "html",
    "section_start": "## 💻 Software Engineering Roles",
    "section_end": "## Other Roles",
},
  4. For a markdown pipe-table repo (like vansh's), use "parser": "markdown" instead.
  5. The hard part is finding correct section_start and section_end markers (see "Find section markers" tutorial below).
  6. Commit and push the workflow change.
  7. Trigger a workflow run manually (Actions tab → Watch files → Run workflow). The first run will bootstrap the new source silently — no alerts for existing rows.
  8. Add the matching entry to WATCHERS in web/app.py too so the dashboard shows the new source.
  9. Refresh the dashboard. The new source should appear under "Watched job repos" with its current SHA and row count.
Find section markers for a new repo

Section markers are exact-match strings that bound the area of the README the bot parses. They prevent us from alerting on non-SWE roles, inactive listings, etc.

  1. Open the upstream repo's README on GitHub (the file you want to watch).
  2. Switch to Raw view to see the source markdown.
  3. Identify the section heading just before the SWE table starts. Use that exact text (including emoji, capitalization) as section_start. Example: ## 💻 Software Engineering Internship Roles
  4. Identify the next heading or marker AFTER the SWE section. Use that as section_end. The bot stops parsing at this marker. Example: <summary>🗃️ Inactive roles (Simplify uses this to fence off inactive jobs).
  5. If you can't find good markers, you can use the start of the next section heading (e.g. ## Data Science) as section_end — but be precise so you don't accidentally include extra rows.
  6. After adding the watcher, do a manual workflow run and check the log for Bootstrapped … with N rows. If N is way off from reality, your markers are wrong.
Re-bootstrap a watched source (forget all its rows)

Useful when: you changed the parser config and want a clean slate, or the row snapshot got corrupted.

  1. Clone or pull the latest job-watch repo locally.
  2. Open .watcher_state.json in a text editor.
  3. Find the entry for the source you want to reset, e.g. "SimplifyJobs/Summer2026-Internships": { "last_sha": "...", "rows": [...] }.
  4. Delete that entire entry (including the surrounding commas as needed to keep JSON valid).
  5. Commit and push.
  6. The next workflow run will see the source as missing in state and re-bootstrap silently (logs Bootstrapped REPO at SHA with N rows.). No spurious alerts.
If you delete the file entirely, ALL watched sources re-bootstrap on the next run.
Remove a watched repo
  1. Edit watch-files.yml → delete the entry from the WATCHERS list.
  2. Also delete the corresponding entry from WATCHERS in web/app.py.
  3. Optionally clean up .watcher_state.json by removing that source's entry (it's harmless if left — just dormant data).
  4. Commit and push. Next run won't poll that source anymore.
