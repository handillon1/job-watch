Set up cron-job.org from scratch

cron-job.org hits GitHub's workflow_dispatch API every minute to bypass GitHub's flaky internal cron. This is what makes the apply-button reliable.

  1. First, create a GitHub PAT (see next tutorial below).
  2. Sign up at cron-job.org (email + password, no credit card).
  3. From the dashboard, click Create cronjob.
  4. Fill in:
    • Title: job-watch trigger
    • URL: https://api.github.com/repos/Mekski/job-watch/actions/workflows/watch-files.yml/dispatches
    • Schedule (Common tab): Every minute
  5. Switch to the Advanced tab:
    • Request method: POST
    • Request headers (add three):
      • Authorization → Bearer <YOUR_PAT>
      • Accept → application/vnd.github+json
      • Content-Type → application/json
    • Request body: {"ref":"main"}
  6. Click Create cronjob. Within ~60 seconds new runs should appear in the GitHub Actions tab.
Create the GitHub PAT for cron-job.org
  1. Open Settings → Developer settings → Personal access tokens → Fine-grained tokens → Generate new token.
  2. Token name: cron-job-org-trigger (or whatever helps you identify it later).
  3. Expiration: 1 year is the longest fine-grained tokens allow. You'll need to renew it then.
  4. Repository access: choose Only select repositories → pick Mekski/job-watch (limit blast radius).
  5. Repository permissions: scroll to Actions → set to Read and write. Leave everything else at "No access."
  6. Click Generate token. Copy it immediately — GitHub only shows the token value once. Looks like github_pat_....
  7. (Optional) Verify before continuing: 
    curl -X POST \
  -H "Authorization: Bearer YOUR_PAT" \
  -H "Accept: application/vnd.github+json" \
  -d '{"ref":"main"}' \
  https://api.github.com/repos/Mekski/job-watch/actions/workflows/watch-files.yml/dispatches
    Expected: empty response, HTTP 204. A new workflow run should appear in the Actions tab within ~5 seconds.
Renew an expired PAT

Fine-grained PATs expire after the period you chose (max 1 year). GitHub emails you a heads-up before expiry. When it expires, cron-job.org starts getting 401 responses and runs stop.

  1. Generate a fresh PAT following the steps above. Same scope: Actions: Read and write on this repo.
  2. Go to your cron-job.org dashboard → click into the job-watch trigger job → Advanced tab → Headers.
  3. Update the Authorization header value with the new Bearer <NEW_PAT>.
  4. Save. The next fire (within ~60s) should succeed. Refresh this dashboard to confirm.
Debug: cron-job.org stopped firing
  1. If this card is red, the most recent GH workflow run is >5 minutes old.
  2. Go to cron-job.org dashboard. Open your job-watch trigger job.
  3. Click the History tab. You'll see each fire and its HTTP response code. Common patterns:
    • 204: success. If you see 204s but the GH dashboard is stale, the issue is on GitHub's side — check githubstatus.com.
    • 401: PAT expired or invalid. Regenerate (see above) and update the Authorization header.
    • 403: PAT exists but lacks Actions: Read and write. Re-check token scopes.
    • 404: URL is wrong. Re-check the URL field matches https://api.github.com/repos/Mekski/job-watch/actions/workflows/watch-files.yml/dispatches.
    • Job paused: check the toggle on the cron-job.org dashboard.
  4. The workflow's built-in schedule: triggers in watch-files.yml are a fallback — they keep the bot working at degraded latency (1 fire/hour-ish) even when cron-job.org is down.
