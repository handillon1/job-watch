Understand pending vs applied

The bot maintains a simple state machine for each job alert:

  1. Pending: a new SWE row was detected, an alert was DM'd to you on Telegram, and the bot is waiting for you to tap "Mark as Applied." The job sits here until tapped.
  2. Applied: you tapped the button. The bot wrote a row to your Google Sheet, edited the original Telegram message to "✅ Logged," and moved the hash from pending to applied with a timestamp.
  3. Neither: you ignore the alert. The hash stays in pending indefinitely (no harm done). If you scroll back and tap the button later, it'll still work as long as the alert message in Telegram still has its button.

The "telegram offset" is internal bookkeeping — the highest update_id the bot has consumed from Telegram's getUpdates queue, used to avoid re-processing the same callback.

Tap an alert and it doesn't process
  1. If you tapped "Mark as Applied" and the loading spinner times out without "✅ Logged" showing, the bot will catch up on its next run (within ~1 min with cron-job.org running).
  2. Check the GitHub Actions card — is the bot actively running? If no run in the last 5 min, check the cron-job.org card.
  3. If the run IS happening but logs show Received 0 update(s): Telegram dropped the callback. Causes: the message's button was already removed (you tapped one that was already processed), OR Telegram's 24h update TTL expired. Re-tap if the button is still visible on the message.
  4. If logs show Sheets append failed: check the Sheets card. Common cause: sheet not shared with SA.
  5. If logs show callback hash X not in pending — already logged or stale: the bot already processed this hash but Telegram re-delivered the callback for some reason. Harmless — the row was already added to the sheet.
Re-tap an alert (force re-processing)
  1. Find the alert in your Telegram chat. Make sure the "Mark as Applied" inline button is still visible (if the message shows "✅ Logged," it's been processed already).
  2. Tap the button. Telegram queues a callback_query event for the bot.
  3. Within ~60 seconds, the next cron-job.org-triggered workflow run will:
    • Poll Telegram for new callbacks
    • Look up the hash in pending
    • Append a row to your sheet
    • Edit the message to "✅ Logged"
    • Move the hash from pending to applied
  4. Refresh this dashboard. The Bot state card should show pending decremented and applied incremented.
Manually fix the state file

If .bot_state.json got corrupted (the bot-state card shows "invalid JSON") or you want to remove a stale pending entry that'll never be tapped:

  1. Clone or pull the latest job-watch repo locally.
  2. Open .bot_state.json in a text editor.
  3. Schema: 
    {
  "telegram": { "last_update_id": <number> },
  "pending": { "<hash>": { "company": "...", "role": "...", ... }, ... },
  "applied": { "<hash>": { ..., "applied_at": "..." }, ... }
}
  4. To remove a stale pending: delete the hash entry from pending. Keep JSON valid (commas, braces).
  5. To move pending → applied manually: cut from pending, add to applied with an applied_at ISO timestamp.
  6. Commit and push. The next run will use the updated state.
If you make a syntax error, the bot will fail to load state and log "invalid JSON". Re-fix and push. Or delete the file entirely to start fresh — the bot will re-create it.
Full reset

Nuclear option: clear all state and start over.

  1. Locally, delete .bot_state.json and .watcher_state.json.
  2. Commit and push.
  3. Trigger a workflow run manually. The bot will:
    • Bootstrap each watched repo silently (snapshot the current rows, no alerts)
    • Initialize .bot_state.json with empty pending/applied
    • Reset the Telegram offset to 0
  4. From here forward, only new SWE rows trigger alerts. You'll lose your applied history (rows in the Google Sheet stay — only the in-bot state is reset).
