Manually trigger a workflow run
  1. Go to Actions → Watch files in external repo.
  2. Top-right of the runs list: click Run workflow.
  3. Make sure the branch is main. Click the green Run workflow button.
  4. A new run appears within ~5 seconds. Click into it to watch it progress.
  5. Useful when: you just changed a secret, you suspect a missed alert, or cron-job.org is paused.
View logs for a specific run
  1. Open the workflow runs list.
  2. Click any run to open it. You'll see two jobs: watch and process_applies.
  3. Click a job to expand it. The detailed log lives under Run watcher (or Process callbacks for the apply job).
  4. Helpful log lines to look for:
    • Bootstrapped … = first time seeing a source, no alerts sent
    • No new listings. = SHA changed but no new SWE rows
    • Sent N job alert(s). = alerts went out
    • Received N update(s) = N callback queries pulled from Telegram
    • logged HASH → COMPANY / ROLE = sheet append succeeded
Diagnose a failed run
  1. Open the failed run in the Actions tab. Failures usually surface as red ❌ on a specific step.
  2. Common failures and fixes:
    • HTTP 500 on git push: transient GitHub outage. Workflow retries 5x; if all fail, ignore — the next run will succeed.
    • HTTP 403 on Sheets append: service account isn't shared on the sheet. Check the GCP card → Share with SA tutorial.
    • HTTP 401 on Telegram: bot token is invalid. Re-create or check the secret value.
    • HTTP 409 on getUpdates: webhook is set on the bot. See the Telegram card → "Disable a stuck webhook" tutorial.
    • WARNING: extracted 0 rows: section markers in the watched README have drifted (e.g. upstream renamed the heading). See the Watched job repos card → "Find section markers".
  3. If a run failed at the commit step but the actual work succeeded, no action needed — the state file was already updated and pushed earlier in the run.
Add a new GitHub secret
  1. Go to Settings → Secrets and variables → Actions.
  2. Click New repository secret.
  3. Name: use SCREAMING_SNAKE_CASE (e.g. NEW_API_KEY). Value: paste the secret.
  4. Click Add secret.
  5. Reference it in the workflow YAML as {% raw %}${{ secrets.NEW_API_KEY }}{% endraw %} inside an env: block on the step that needs it.
  6. Re-trigger the workflow (manually or wait for cron) to pick up the new env var.
